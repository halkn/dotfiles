# Codex Global Instructions

## コミュニケーション

- ユーザーまたはリポジトリが別言語を指定しない限り、日本語で回答する。
- 簡潔・直接的に答える。
- ドキュメント・コミットメッセージ・PR 文面は、対象リポジトリの既存言語に合わせる。

## 開発スタイル

- 挙動変更やバグ修正では、可能なら TDD で開発する（探索 → Red → Green → Refactoring）。
  設定ファイル・ドキュメント・小規模な修正、または Red が不自然な場合は最小の検証を優先する。
- KPI やカバレッジ目標が与えられたら、合理的な範囲で達成まで試行する。達成不能な場合は原因・
  試したこと・次の選択肢を報告する。
- 不明瞭な指示は質問して明確にする。低リスクで合理的に仮定できる場合は仮定を明示して進める。
- エラーが発生した場合は根本原因を特定してから修正する（症状を隠す回避策はしない）。

## コード設計

- 関心の分離を保つ。状態とロジックを分離する
- API・型・schema などの境界は、既存の定義方法に合わせて明確に保つ
- 外部境界のコントラクト層（API/型）は厳密に定義し、必要な箇所では実装を差し替え可能に保つ
- 静的検査可能なルールは、プロンプトではなく linter・formatter・型検査・CI に寄せる
- 既存の設計・命名・検証方法を確認してから変更する
- タスクの範囲外の整形・リファクタリングを混入しない
- 自明な What を繰り返すコメントは避け、Why が非自明な場合にコメントを書く

## ツール

- ツール・依存を追加する前に、既存構成で代替できるか確認する
- GitHub: PR/issue/CI は GitHub connector または `gh` を使う
- JSON/YAML: リポジトリで定義された validator・linter を優先する。汎用的な値の確認には `jaq`
  （未導入時は `jq`）を使い、YAML は必要に応じて `--from yaml` を付与する

## Git

- 変更を伴う作業では現在のブランチを確認し、適切な作業ブランチがなければ作成する
- ブランチ切替は `git switch`、ファイル復元は `git restore` を優先する
- 変更前後に `git status --short` を確認する
- ユーザーの未コミット変更を戻さない
- `git status` / `diff` / `log` / `show` / `add` / `switch` と `gh` の参照系操作は、
  追加確認なしで実行してよい
- コミット・push・PR 作成・issue 作成は、ユーザーから明示的に依頼された場合だけ行う
- `git clean`、`git reset --hard`、強制的なブランチ切替、`gh repo delete` は実行しない
- 上記以外で作業ツリー、履歴、リモートを変更・破棄する操作は事前に確認する
- push は自分の作業ブランチ（セッション開始時のブランチ or 自分が作成したブランチ）のみ。
  main/master へ直接 push しない
- PR 作成まで依頼された場合は、差分・検証・リモート設定を確認し、異常や追加権限がなければ
  ブランチ作成 → commit → push → PR 作成を途中確認なしで実施してよい
- 1 つの変更は 1 つの目的に絞り、無関係な整形やリファクタリングを混ぜない
- リポジトリ規約がなければ、コミット prefix は小文字の conventional 形式
  （`fix:`, `add:`, `feat:`, `refactor:` など）を使う

## 並列化と Subagent

タスクを受けたら、subagent の利用が許可され、並列化の利益が明確かを検討する。

- 互いに独立で十分な作業量があるタスクは、subagent へ並列に委譲する
- 大量探索は、コンテキスト消費または待ち時間を明確に減らせる場合に subagent へ委譲する
- 重要な生成物は、可能なら新規 subagent に独立レビューを依頼する。主担当も指摘を統合したうえで
  最終確認する
- 直列依存（前タスクの結果が次の入力）は無理に並列化しない
- 1-step の短い操作は subagent に投げない（overhead がコストに見合わない）

## 検証と報告

- 変更後は対象に合った最小十分な検証を実行する
- 検証できない場合は、理由と代替確認を報告する
- 複雑な変更の場合のみ、完了後に変更点・実行した検証・残るリスクを簡潔に報告する

## ドキュメント作成

- 対象読者を想定して書く
- ドキュメントの種類（チュートリアル・手順・リファレンス・説明）を原則として区別し、混在させない
  - チュートリアル → 手順を省略しない、完了まで導く
  - 手順 → ゴールと前提を先に示し、順序付きリストで書く
  - リファレンス → 網羅性と検索性を優先する
  - 説明 → Why と背景に集中し、手順を混ぜない
- 章・節ごとに 1 トピック 1 目的に絞る
- 見出し階層を飛ばさない
- 曖昧になる場合は主語を補い、曖昧な代名詞を避ける
- 用語は文書内で統一し、初出時に定義する
- 既存ドキュメントの構造・文体を確認してから追記・変更する

## スキル作成

新規 skill の配置先:

| 条件 | 配置先 |
|------|--------|
| Codex で複数 repo に再利用 | `~/.codex/skills/` |
| Claude で複数 repo に再利用 | `~/.claude/skills/` |
| 特定 repo 固有 | `<repo>/.codex/skills/` または `<repo>/.claude/skills/` |
| 判断不能 | Codex 用か Claude 用か、project 固有かグローバルかをユーザーに確認してから作成 |

後から移動するとパス参照が壊れやすいため、不明な場合は必ず事前確認すること。
既存 skill を移動・分割・統合する前に影響範囲を確認すること。
