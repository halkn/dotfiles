# Global Instructions

## コミュニケーション

- 常に日本語で回答する。
- 簡潔・直接的に答える。

## 開発スタイル

- TDD で開発する（探索 → Red → Green → Refactoring）。ただし、設定ファイル・ドキュメント・小規模な修正、または Red が不自然な場合は最小の検証を優先する。
- KPI やカバレッジ目標が与えられたら達成するまで試行する。達成不能な場合は原因・試したこと・次の選択肢を報告する。
- 不明瞭な指示は質問して明確にする。低リスクで合理的に仮定できる場合は仮定を明示して進める。
- エラーが発生した場合は根本原因を特定してから修正する（症状を隠す回避策はしない）。

## コード設計

- 関心の分離を保つ。状態とロジックを分離する
- API・型・schema などの境界は、既存の定義方法に合わせて明確に保つ
- コントラクト層（API/型）は厳密に定義し、実装層は差し替え・再生成可能に保つ
- 静的検査可能なルールは、プロンプトではなく linter・formatter・型検査・CI に寄せる
- 既存の設計・命名・検証方法を確認してから変更する
- タスクの範囲外の整形・リファクタリングを混入しない
- コードコメントは Why が非自明な場合のみ書く（What は書かない）

## アーキテクチャ設計

新機能の設計・レビュー・大規模リファクタリング時に適用する。小規模な変更や設定ファイル修正では省略してよい。

- 設計を提案する前に、問題ドメインの境界・変化点・不変条件を整理する
- 依存の方向（外→内）を確認し、逆方向があれば指摘する（ビジネスロジックは外部依存を知らない）
- 変更頻度の高い部分と安定した部分を分け、頻繁な部分は差し替えやすく設計する
- 横断的関心事（認証・ロギング・エラー処理）の方針を早期に確認し、既存の一貫性を保つ
- 長期運用が前提のプロダクトでは、重要な設計決定を ADR として記録する（短く・単一決定・Markdown・追記専用）

## ドキュメント作成

README・手順書・API ドキュメントなど一般的な技術文書を書く際に適用する。

- 各セクションを Concept / Task / Reference / Troubleshooting のいずれかに分類して書く（CTRT）
  - Concept: 「なぜ・何か」の説明のみ（手順を混入しない）
  - Task: 「どう使うか」の手順。タイトルは「動詞 + 名詞」形式
  - Reference: 参照情報のみ。Troubleshooting: 問題対処のみ
- Why（判断の根拠）と How（操作手順）を優先する。What（構造説明）は最小限に
- コードで表現できる情報はドキュメントに書かない（型・命名・テストで伝える）
- ドキュメントはコードと同一リポジトリに置き、変更と同一 PR に含める
- source of truth は1つに保つ。重複文書を作らず、既存があれば参照する
- 図はテキスト管理可能な形式（Mermaid 等）を優先する

## 設計ドキュメント

新機能・システム変更の設計判断を記録する文書。小規模な変更では省略してよい。

- コードを書く前に、設計判断とトレードオフを散文で整理する
- 「説明・定義できないものは設計できていない」を前提に、書く過程で設計を再評価する
- 必須セクション: 文脈 / Goals / Non-goals / 提案する設計 / 検討した代替案 / トレードオフ / 横断的関心事
- Non-goals と代替案は省略しない（「なぜこの案か」を先回りで潰す）
- 推奨案を1つに断定せず、トレードオフを提示する（判断は人間が行う）
- 出荷後の変更は元文書へのリンクつき改訂（amendment）方式で運用する
- 不確実な情報には確信度を明示し、可能なら公式 URL を添付する

## ツール

- ツール・依存を追加する前に、既存構成で代替できるか確認する
- GitHub: PR/issue/CI は `gh` を使う
- YAML: 値の確認は `yq`（Go 版 mikefarah/yq）、構文は `mise run lint`（`python -c "import yaml"` は使わない）

## Git

- 作業前に必ずブランチを作成する
- ブランチ切替は `git switch`、ファイル復元は `git restore` を使う（`git checkout` は使わない）
- 変更前後に `git status --short` を確認する
- ユーザーの未コミット変更を戻さない
- 破壊的操作（`git reset --hard`、強制切替、削除など）は事前に確認する
- コミット・push・PR 作成は明示的に依頼された場合だけ行う
- 1 つの変更は 1 つの目的に絞り、無関係な整形やリファクタリングを混ぜない
- コミット prefix は小文字の conventional 形式: `fix:`, `add:`, `feat:`, `refactor:` など

## 並列化と Subagent

タスクを受けたら「並列化できる subtask は何か」「subagent に投げてメインコンテキストを空けられるか」を最初に検討する。

- 互いに独立な 2 つ以上のタスク → Agent tool で 1 メッセージ内に並列 dispatch
- 大量探索・3 クエリ超の調査 → Explore/general-purpose subagent に委譲、メインは要約のみ受け取る
- 自分の生成物の bias-free 評価 → 新規 subagent（自分で再読は禁じ手）
- 直列依存（前タスクの結果が次の入力）は無理に並列化しない
- 1-step の短い操作は subagent に投げない（overhead がコストに見合わない）

## 検証と報告

- 変更後は対象に合った最小十分な検証を実行する
- 検証できない場合は、理由と代替確認を報告する
- 複雑な変更の場合のみ、完了後に変更点・実行した検証・残るリスクを簡潔に報告する

## スキル作成

新規 skill の配置先:

| 条件 | 配置先 |
|------|--------|
| 複数 repo で再利用 | `~/.claude/skills/` |
| 特定 repo 固有 | `<repo>/.claude/skills/` |
| 判断不能 | ユーザーに確認してから作成 |

後から移動するとパス参照が壊れやすいため、不明な場合は必ず事前確認すること。
既存 skill を移動・分割・統合する前に影響範囲を確認すること。
